This is the main `@lmnr-ai/lmnr` SDK package — OpenTelemetry-based tracing for the Laminar observability platform.

# Debug mode / replay (`src/debug/`)

- A "debug run" is just a normal process started with `LMNR_DEBUG*` env vars; there is no cache server, SSE, orchestrator, entrypoint flag, or bundling anymore (all removed from `lmnr-cli` + `@lmnr-ai/lmnr`). The replay cache lives in-process.
- `src/debug/` is a **cross-language parity surface** with Python `src/lmnr/sdk/debug/`. Keep `config.ts`, `spine.ts`, `replay-cache.ts`, `pointer.ts`, `replay.ts`, `source-trace.ts`, `index.ts` line-comparable. The shared test vectors (`test/data/debug/*.json`) are byte-identical copies of `lmnr-python/tests/data/debug/`; change them in lockstep across both repos.
- Parity invariants that MUST match Python exactly: truthy set `["true","1","yes","on"]`; `replayEnabledForConfig = replayTraceId !== null && cacheUntil > 0`; pointer key order `{trace_id, session_id, replay_trace_id, cache_until, debugger_url, started_at}`; `CONSOLE_PREFIX = "LMNR_DEBUG_RUN "`; spine = shallowest looping path / tie-break earliest start / fallback shallowest single call; `ReplayCache` truncates `payloads.slice(0, cacheUntil)`.
- The pointer's **`started_at` is captured at `DebugRuntime` construction (SDK init), NOT inside `buildPointer`/`emitPointer`**. The pointer is emitted at shutdown, so reading the clock in `buildPointer` would stamp `started_at` near process exit rather than run start. `DebugRuntime` records `_startedAt = new Date().toISOString()` as a field initializer and passes it through to `buildPointer({ startedAt })`; the `startedAt` arg defaults to now only for standalone callers (the pointer unit tests). Python mirrors this with `DebugRuntime._started_at` → `build_pointer(started_at=...)`.
- **`replayEnabled()` gates on `getRuntime()?.replayConfigured`, NOT on `getRuntime() !== null`.** A debug-no-replay run (`LMNR_DEBUG` set but no `LMNR_DEBUG_REPLAY_TRACE_ID` / `LMNR_DEBUG_CACHE_UNTIL`) still creates a runtime — it's needed to stamp `rollout.session_id` and emit the run pointer — but replay must stay off there, or every LLM call resolves a span path and advances a per-path occurrence counter against a cache that will never be built. `replayConfigured` is `replayEnabledForConfig(this._config)` (`replayTraceId !== null && cacheUntil > 0`); crucially it stays true during the async cache-load window (`_cache === null` but replay IS configured), so the counter still advances there as designed. Python mirrors this with `DebugRuntime.replay_configured` (`self._config.replay_enabled`) — keep both in lockstep. Python has no async-fill window, so its getter is purely for line-parity.
- The per-path **occurrence counter lives on `DebugRuntime` (`_counters`), NOT on `ReplayCache`**. `DebugRuntime.getCached` advances the counter unconditionally — even while `_cache === null` — then consults the cache only if present. This matters because `initDebugRuntime` fills the cache asynchronously (`setCache`) after returning, so spine calls in the loading window would otherwise run live without consuming a slot and misalign record-vs-replay after the cache lands. `ReplayCache.getCached(spanPath, occurrence)` is now a pure lookup with no internal counter; Python mirrors this design for line-parity even though its cache builds synchronously.
- `Laminar.initialize()` is synchronous but the source-trace fetch is async: `initDebugRuntime(client, debuggerUrl)` returns `{ runtime, ready }`. The runtime registers immediately (so `sessionId` is available to stamp `rollout.session_id` on trace metadata) and the cache fills in the background via `runtime.setCache`. In tests, `await ready` before asserting on cached payloads. Any fetch/build failure degrades to debug-no-replay (warn, never throw).
- `initDebugRuntime` is one-shot (the module-level `initialized` flag makes it idempotent), but `Laminar.shutdown()` supports a later `initialize()`, so it MUST call `resetDebugRuntime()` (alongside the other state resets) to clear `runtime`/`initialized` — otherwise a second `initialize()` resurrects the first run's stale cache / session metadata / spent pointer and never re-reads `LMNR_DEBUG*`. `resetDebugRuntime` is the public reset (tests use it too); Python mirrors this with `reset_debug_runtime` called from its `shutdown()`.
- **The debug `exit` pointer hook must be removed on `shutdown()` or it leaks across init/shutdown cycles.** `_initDebugRuntime` registers `process.once("exit", () => runtime.emitPointer())` so the run pointer is emitted at process teardown. But `process.once` only auto-detaches *after* `exit` actually fires — a repeated `initialize()` (or an init/shutdown loop, common in test suites) re-registers without ever firing, piling up listeners until Node logs `MaxListenersExceededWarning` (~10). Fix: keep the handler by reference in a static `Laminar.debugExitHook`, `removeListener` any prior hook before registering a new one, and `removeListener` + null it in `shutdown()` (the pointer is already emitted synchronously at the top of `shutdown()`, so the hook there is a redundant no-op anyway). `laminar.ts` is NOT part of the `src/debug/` line-parity surface, so this listener-management divergence from Python doesn't break parity. (Python registers `atexit.register(runtime.emit_pointer)` per init without unregistering, but `emit_pointer` is one-shot per runtime so stale handlers become harmless no-ops; if Python init/shutdown leak ever becomes a concern, mirror this cleanup with `atexit.unregister`.) Tested in `test/initialize.test.ts` (`does not leak exit listeners across init/shutdown cycles`).
- The in-flight cache-build `ready` promise is held at module scope (`readyPromise`) so the idempotent branch returns the SAME promise as the first call, NOT a fresh `Promise.resolve()`. A second `initDebugRuntime` while the first build is still loading would otherwise hand the caller an already-settled promise, letting replay lookups run before `setCache` lands. `resetDebugRuntime` resets `readyPromise` to `Promise.resolve()` too. This is TS-only — Python builds the cache synchronously inside `init_debug_runtime`, so it has no async-fill window and no `ready` promise to track.
- **A debug run registers its SDK-minted session id with the backend** via `client.rolloutSessions.register({ sessionId })` (`POST /v1/rollouts/{sessionId}`, body `{ name }`, project-API-key auth, in `@lmnr-ai/client`'s `RolloutSessionsResource`). Without it a bare `LMNR_DEBUG=true` run mints a session id the backend never learns about, so the run is invisible in the UI. The call lives in `Laminar._initDebugRuntime` right after the `runtime === null` guard and is **fire-and-forget**: `void client.rolloutSessions.register(...).then(...).catch(...)` — `initialize()` is synchronous, so registration must never block or throw into it (Python mirrors this as a best-effort blocking call since it has no event loop). `register` returns `Promise<string | null>` — the backend-resolved `projectId` (`body.projectId ?? null`, `null` on parse failure) — so the `.then` can `logger.info` the human-facing debugger URL (TS `initializeLogger` already defaults to colored `info`). The full-URL construction is **one shared code path**, `DebugRuntime.debuggerSessionUrl()` — used by BOTH the `.then` console line AND the run pointer's `debugger_url` field (the pointer is emitted at shutdown, so the runtime must remember the project id). The `.then` calls `runtime.recordProjectId(projectId)` (first-wins, mirrors `recordTraceId`) then `logger.info(\`Laminar debugger session: ${runtime.debuggerSessionUrl()}\`)`; `debuggerSessionUrl()` returns the full `${base}/project/${projectId}/debugger-sessions/${sessionId}` (the pointer would otherwise store only the base `LMNR_FRONTEND_URL`, useless to the coding agent reading `last-run.json`), falling back to the bare base `debuggerUrl` when no project id was recorded (register failed / fire-and-forget hasn't resolved) and `null` when even the base is unset. Python mirrors this with `DebugRuntime.debugger_session_url()` / `record_project_id` — keep in lockstep. After editing `register`'s return type you MUST `pnpm --filter @lmnr-ai/client build` before linting `packages/lmnr` — the consuming code reads the *built* `@lmnr-ai/client` types, and a stale `void` return makes `projectId` infer as `never` (lint error `restrict-template-expressions`). The backend endpoint is an idempotent upsert keyed on the SDK-supplied id; a null/omitted `name` never clobbers a name set elsewhere, and now returns JSON `{sessionId, projectId}` (it used to be an SSE stream — see `lmnr/CLAUDE.md`). The old SSE run-command stream and PATCH span-update endpoints were removed in the rework — only `register` (POST) and `delete` (DELETE) survive on the resource. Tested in `packages/client/test/rollout-sessions-resource.test.ts`.
- **`LMNR_DEBUG_FROM_LAST_RUN` (truthy) seeds the config from the previous run's pointer file** instead of requiring the ids be copied into env vars. When set, `buildDebugConfig` calls `loadLastRun()` (reads `${CWD}/.lmnr/last-run.json`, best-effort `{}` on any error) and maps the file's `trace_id` → this run's `replayTraceId`, reuses its `session_id` and `cache_until`. Individual `LMNR_DEBUG_*` env vars still override per-field. `loadLastRun` imports `POINTER_DIR`/`POINTER_FILE` from `pointer.ts`. Keep line-comparable with Python `_load_last_run` in `config.py`. Tested in `test/debug/config.test.ts`.
- Naming: new env/file/module naming is `debug`; the persisted metadata key stays `rollout.session_id` (intentional — do not rename to `debug.session_id`).
- `source-trace.ts` fetches in two phases over `LaminarClient.sql.query`: phase 1 SELECTs `path, span_type, start_time, end_time` (identify it by the `"span_type, start_time"` substring when faking the SQL client in tests); phase 2 SELECTs payloads for the chosen spine path only.
- **A missing / null `end_time` maps to `Infinity`, NOT `0`** (`toEpoch(value, missingDefault)` — `Infinity` for `end_time`, default `0` for `start_time`). The §F overlap guard is `cur.startTime < prev.endTime`; collapsing an unknown end to `0` would make that comparison always false and let overlapping spine calls pass the guard, so replay would proceed when it should force a live run. `Infinity` makes an unknown end conservatively overlap. Python `_to_epoch(value, missing_default=inf)` mirrors this — keep both in lockstep.
- **`toEpoch`'s lexical fallback (in `source-trace.ts`) must preserve chronological order.** When `Date.parse` fails on an ISO-8601 string, the fallback score must be a *fractional positional* encoding (`score += Math.min(c.charCodeAt(0),255) * weight; weight /= 256` per char, earliest char weighted most) — a flat sum of char codes is position-insensitive and silently reorders the spine (a later timestamp can score lower than an earlier one). Float64 only resolves the first ~6-7 chars; closer strings tie (never reverse), and the SQL `ORDER BY start_time` + stable sort preserve original order on a tie. `Date.parse` already accepts nanosecond ISO strings (truncating to ms), so TS needs no parse-normalization step — but Python's `_to_epoch` DOES (its `datetime.fromisoformat` rejects >6 fractional digits pre-3.11); keep the fallback identical across both.
- **The run pointer's `trace_id` is recorded from two places**, because not every debug run opens a root span. The normal path is `LaminarSpanProcessor.onStart` → `recordDebugTraceId` when `parentSpanId === undefined`. But a run attached via `LMNR_SPAN_CONTEXT` pushes a wrapped (non-recording) parent span onto the context (`_initializeContextFromEnv`), so EVERY span the process creates has a defined parent and the root-span hook never fires — the pointer would emit an empty `trace_id`. `_initializeContextFromEnv` therefore also calls `getRuntime()?.recordTraceId(otelTraceIdToUUID(otelSpanContext.traceId))` after pushing the context (all spans in the run share that trace id; `recordTraceId` keeps the first). `_initDebugRuntime` runs before `_initializeContextFromEnv` in `initialize()`, so the runtime is already registered at attach time. Python mirrors this in `_initialize_context_from_env` (`Laminar._record_debug_trace_id_from_env`).
- **The debug run's `rollout.session_id` reaches spans via global metadata, which `LaminarSpanProcessor.onStart` stamps on EVERY span.** Unlike Python, TS does not carry global metadata on the OTEL context — `setGlobalMetadata` only stores a static field on `LaminarContextManager`. So `onStart` must read `getGlobalMetadata()` and apply it via `metadataToAttributes(...)` directly; relying on the `ASSOCIATION_PROPERTIES_KEY`-on-context block alone would only stamp spans created under `observe` / `startActiveSpan` (those merge global metadata in via `buildContextProperties`), and pure auto-instrumentation traces would silently lose `rollout.session_id`. The global-metadata stamp runs BEFORE the association-properties block so per-span metadata (which already includes the global metadata) still wins. Python gets this for free because `initialize()` attaches `__global_metadata` onto the ambient OTEL context, which its processor reads via `get_value(CONTEXT_METADATA_KEY)`.

# Context / startActiveSpan

- `Laminar.startActiveSpan` activates the span on `LaminarContextManager`'s AsyncLocalStorage stack. Sibling / unrelated async tasks (i.e. callbacks that run outside the returned promise's chain — `setImmediate`, separate timers, independent processor callbacks, etc.) do NOT inherit the activated span — they live in a different ALS frame. For spans that must parent across detached async callbacks, pass `global: true`: the span is pushed onto a process-global stack that `getContext()` falls back to when the ALS stack is empty. The span is popped from that stack automatically in `LaminarSpan.end()`. Use sparingly — it is roughly analogous to Python's `context.attach(...)` / `start_as_current_span` global activation.
- If you pass `context: ...` into `startSpan` / `startActiveSpan` it replaces the usual `LaminarContextManager.getContext()` base — so if you need the global fallback, build your context on top of `LaminarContextManager.getContext()` rather than on `otelContext.active()`.
- `LaminarSpan.end()` only calls `popContext()` once (for the activated-span push). Do NOT push additional context frames elsewhere in `_startSpan` without a matching pop — `popContext`'s filter drops _already-ended_ spans by spanId, but a leaked wrapper of a still-live parent span (e.g. `trace.wrapSpanContext(...)` built from `parentSpanContext`) looks "valid" and stays on the stack, growing it unboundedly across repeated start/end pairs. The `parentSpanContext` path is resolved purely via `trace.setSpan(entityContext, ...)` + passing `entityContext` to `getTracer().startSpan(..., entityContext)` — no push needed.

# OpenTelemetry Instrumentations

- Instrumentations live under `src/opentelemetry-lib/instrumentation/<name>/` and are registered in `src/opentelemetry-lib/tracing/instrumentations.ts` (both auto-registration path and the `manuallyInitInstrumentations` path — keep the two in sync).
- For manual instrumentation, expose an option key on `InitializeOptions.instrumentModules` and wire it through `manuallyInitInstrumentations`.
- `TracingProcessor` from `@openai/agents` requires the interface methods return `Promise`. Most of our methods have no `await` — add a file-level `/* eslint-disable @typescript-eslint/require-await */` around the class rather than sprinkling `await Promise.resolve()`.
- `@openai/agents` re-exports types and classes from `@openai/agents-core` and `@openai/agents-openai`. Import from `@openai/agents` to avoid adding `-core`/`-openai` as direct dependencies.
- When patching prototypes registered via multiple `InstrumentationNodeModuleDefinition` entries that resolve to the same class (e.g. `@openai/agents` and `@openai/agents-openai` both expose `OpenAIResponsesModel.prototype`), guard `_wrap` with a `WeakSet` of already-wrapped prototypes — `shimmer` happily double-wraps and will run your wrapper logic twice per call.
- `AsyncLocalStorage.run(value, fn)` restores the previous context as soon as `fn` returns — for async iterables/streams, wrap **each** `next()`/`return()`/`throw()` call in its own `.run()` so the context is visible across yields, not just at iterator creation.

# Stagehand v3 Instrumentation (`src/browser/stagehand/v3/`)

- `page.sendCDP(method, params)` was added in `@browserbasehq/stagehand` **3.0.2** (2025-11-14). Only **3.0.0** (2025-10-29) and **3.0.1** (2025-10-31) lack it — on those versions Page's `mainSession` is private and there is no `sendCDP`, so CDP calls must go through `page.getSessionForFrame(page.mainFrameId()).send(method, params)` (both methods are public on 3.0.0/3.0.1 Page). Use the `sendCDP(page, method, params)` helper in `cdp-helpers.ts` — it probes `page.sendCDP` first, falls back to `getSessionForFrame`, and throws if neither exists. Without this fallback, recording injection silently fails on 3.0.0/3.0.1 with `page.sendCDP is not a function` and no rrweb events reach the backend even though spans still flow. Stagehand PR #983 (merged pre-3.0.2) made `sendCDP` public specifically for external CDP-based instrumentations like ours; the v3 instrumentation itself landed Dec 2025 when 3.0.5 was latest, so 3.0.0/3.0.1 are the only versions that were ever shipped without the needed API.
- The `@lmnr/browserbase-stagehand-instrumentation` instrumentation declares `['>=3.0.0']` as the supported range, so 3.0.0, 3.0.1, 3.0.2+, and 3.1.x+ must all keep working — test any CDP-touching change against at least one pre-sendCDP version (3.0.1) and one post-sendCDP version (3.1.0).

# AI SDK v7 Telemetry (`src/opentelemetry-lib/instrumentation/aisdk/v7.ts`)

- AI SDK v7 replaced the `experimental_telemetry: { tracer }` wire-a-tracer API with a callback-based `Telemetry` integration (16 optional lifecycle callbacks + `executeTool`) passed as `telemetry: { integrations: [...] }`. Both `telemetry` and `experimental_telemetry` keys are accepted on the call site, but ONLY v7 understands `integrations`; v6 still expects a `tracer`. Detect v7 by `typeof ai.registerTelemetry === 'function'` (added only in v7) and branch `wrapAISDK` accordingly — don't sniff by version string. On v7 the wrapper MUST NOT pass `tracer`, and on v6 it MUST NOT pass `integrations`.
- Span hierarchy is `operation (ai.generateText|ai.streamText|ai.generateObject|ai.streamObject)` → `step (ai.step N)` → flat siblings `ai.llm <model>` and `ai.tool <name>`. Tool spans are NOT children of the LLM span even though tool calls are emitted from LLM content — they're siblings under the same step. Embed/rerank (`ai.embed`, `ai.embedMany`, `ai.rerank`) get a SINGLE LLM span at the operation level with no step/llm children; branch on `operationId` inside `onStart` to set `SPAN_TYPE = LLM` directly. Attribute strategy is hybrid: emit both the legacy `ai.*` keys (`ai.prompt.messages`, `ai.response.text`, `ai.response.toolCalls`) AND OTel GenAI semconv (`gen_ai.system`, `gen_ai.request.model`, `gen_ai.response.id`, `gen_ai.response.finish_reason`, `gen_ai.usage.input_tokens`, `gen_ai.output.messages` with `thinking`/`text`/`tool_call` parts). Laminar's backend parser uses the `gen_ai.*` side, but existing dashboards / other exporters still read `ai.*`.
- Sub-agent nesting (nested `generateText` inside `executeTool`) requires propagating the tool span as parent through BOTH OTel's standard context AND Laminar's ALS: wrap the user's `execute` with `contextApi.with(toolSpanCtx, () => LaminarContextManager.runWithIsolatedContext([...currentStack, toolSpanCtx], execute))`. OTel-aware instrumentations read `contextApi.active()`; Laminar's own `startActiveSpan` only reads its ALS stack — miss either and the inner operation span parents to the outer operation instead of the tool span, breaking sub-agent traces.
- Per `wrapAISDK()` call create EXACTLY ONE `laminarTelemetry()` integration and share it across all wrapped AI functions — v7 dedupes integrations by reference, and a fresh integration per call would reset the internal `callId → spanRef` maps between `generateText` invocations in the same process. `ToolExecutionEndEvent` is a discriminated union on `toolOutput.type`: `"tool-result"` → set `ai.toolCall.output`; `"tool-error"` → `span.setStatus({ code: ERROR })` + `ai.toolCall.error = String(error)`.
- `toolByCallId` is keyed by `toolCallId` (AI SDK guarantees uniqueness per run), NOT by `${callId}:...` like `stepByKey` / `llmByKey`. Because of that, `onFinish` / `onError` cleanup sweeps CANNOT use `key.startsWith(prefix)` to scope tool spans to a single operation — `ToolState` must carry the owning `callId` so the sweep can filter by `tool.callId === targetCallId`. Without this, an `onError` mid-tool-execution leaves the tool span open indefinitely (unterminated span in the backend + map leak).
- In `onFinish` and `onError`, defensive cleanup of orphan child spans (llm / step / tool) must run BEFORE `op.span.end()`. Otherwise the parent operation span gets a stale end timestamp and the children end afterwards — breaking trace hierarchy semantics where every child must end before its parent. `onError` already did this; `onFinish` was ending the op first. The cleanup order within the child sweep also matters: tool spans are children of step spans (see `onToolExecutionStart` where `parentCtx = step?.ctx ?? op.ctx`), so the sweep order must be `llm + tool (leaves under step) → step → op`, NOT `llm → step → tool → op`. A test that only validates "children end before operation" does not catch this mid-level ordering — assert tool.endTime ≤ step.endTime explicitly.
- `SPAN_OUTPUT` on the operation span in `onFinish` must fall back to the serialized tool calls when `event.text` is empty — tool-call-only single-step responses otherwise render as empty in the Laminar UI (the backend reads `SPAN_OUTPUT` for the span output panel; `ai.response.toolCalls` alone is invisible). `onLanguageModelCallEnd` already does this fallback at the LLM-span level; mirror it at the operation-span level.
- Sweep ordering alone is NOT enough to keep child.endTime ≤ parent.endTime deterministically. Four rapid synchronous `span.end()` calls inside `endOrphanChildSpansForCallId` read `performance.now()` / hrtime separately, and consecutive readings on the same tick are not strictly monotonic — even when they are, the test's f64 conversion (`sec * 1e9 + nanos` ≈ 1.78e18 today, >1000× past `Number.MAX_SAFE_INTEGER`) rounds both ends to the same bucket and flips the order. Fix: accumulate the maximum child endTime seen so far and explicitly pass it into `step.span.end(latest)` and `op.span.end(latest)`. The value MUST come from the underlying SDK span's `endTime` field (mutated by `end()`), NOT from `LaminarSpan.endTime` — that wrapper copies the field in its constructor before `end()` has set it, so reading it yields the `[0, 0]` sentinel. See the `readSpanEndTime` helper in `v7.ts` which probes `_span.endTime` first and falls back defensively.
- Tests drive the class with synthetic event payloads (see `test/aisdk-v7-telemetry.test.ts`'s `mkStartEvent` / `mkLlmCallStart` / `mkStepEnd` helpers) — no runtime dep on `ai` is needed and no real HTTP cassettes. Span attributes typed as `any` in callback payloads need a file-level `/* eslint-disable @typescript-eslint/no-explicit-any, @typescript-eslint/no-unsafe-argument */` on `v7.ts` rather than per-line disables.
- `recordOutputs: false` gates user-visible _content_ only (`ai.response.text`, `ai.response.toolCalls`, `ai.rerank.ranking`, embedding shape, serialized `SPAN_OUTPUT`). Metadata — `gen_ai.response.finish_reason` / `ai.response.finishReason` and all `applyUsage()` token counts — must be emitted regardless of `recordOutputs`, otherwise cost/analytics dashboards silently lose per-operation totals. The per-step (`onStepFinish`) and per-LLM-call (`onLanguageModelCallEnd`) paths already do this correctly — keep `onFinish` and any new callbacks aligned.
- `onChunk` in AI SDK v7 streams only the lifecycle markers `ai.stream.firstChunk` / `ai.stream.finish` carry `callId` + `stepNumber` — regular content chunks (`text-delta`, `tool-call`, ...) are plain `TextStreamPart` objects with only `type` / `id` / `text` (or equivalent), and carry NO callId either at top level or on the chunk. See AI SDK source `packages/ai/src/generate-text/stream-text.ts` — the dispatcher publishes the lifecycle chunks at the step boundaries and content chunks in between. To accumulate streaming text correctly, latch the active streaming step on `firstChunk`, route intermediate `text-delta` chunks to its LLM span, and clear the latch on `finish` (and on `onFinish` / `onError` cleanup paths to avoid leaks on aborted streams). An integration that requires callId/stepNumber on every chunk silently drops all text-delta chunks for streamText — buffered text ends up empty.
- Because content chunks carry no callId, the "latched active stream" set can hold more than one entry when 2+ concurrent `streamText` / `streamObject` calls share the same `LaminarTelemetry` instance. A LIFO / most-recently-inserted pick misattributes A's chunks to B as soon as B's `firstChunk` fires while A is still active. The correct behavior is to attribute only when the active set has exactly ONE entry — otherwise leave the chunk unattributed. The buffered-deltas path is only a fallback for aborted / errored streams where `onLanguageModelCallEnd` never fires; the normal path reads text from `event.content` which is correctly scoped per call, so declining to attribute ambiguous chunks costs little. The concurrent-stream regression test in `test/aisdk-v7-telemetry.test.ts` pins this behavior — do not replace the size-1 guard with a LIFO pick.
- `onObjectStepStart` / `onObjectStepFinish` fire ONLY from `generateObject` / `streamObject` (which call `model.doGenerate` / `doStream` directly). `onLanguageModelCallStart` / `onLanguageModelCallEnd` fire ONLY from `generateText` / `streamText` via `stream-language-model-call.ts`. The two callback pairs are mutually exclusive per operation — they never both fire for the same `callId`. Confirmed against vercel/ai `packages/ai/src/generate-object/generate-object.ts:352` + `stream-object.ts:584` (no `onLanguageModelCall*` dispatch) vs. `packages/ai/src/generate-text/stream-language-model-call.ts:295` (the only dispatch sites for `onLanguageModelCall*`). Keying the object-gen LLM span under `stepKey(callId, 0)` is therefore collision-free with `onLanguageModelCallStart`'s default `stepNumber = 0` fallback — no separate namespace needed.
- Object-gen LLM spans must carry attribute parity with the `generateText` LLM path — specifically the minimum-set attributes `gen_ai.request.model` (REQUEST_MODEL) + `gen_ai.response.model` (RESPONSE_MODEL) the backend uses for model-routing and cost analytics. `GenerateObjectStepEndEvent.response` (in `vercel-ai/packages/ai/src/generate-object/structured-output-events.ts`) carries `{ id, modelId, timestamp }`, so `onObjectStepFinish` must pull `gen_ai.response.model` from `event.response.modelId` (not `event.modelId`, which is the request model) and `gen_ai.response.id` from `event.response.id`. In `onObjectStepStart` also set the legacy `ai.model.provider` + `ai.model.id` attributes to mirror `onLanguageModelCallStart` — dashboards keyed off those specific attribute names render object-gen spans uniformly with text-gen spans.
- AI SDK v6+ `LanguageModelUsage` moved detail fields under `inputTokenDetails` / `outputTokenDetails`. `applyUsage()` must read both the deprecated top-level fields (`cachedInputTokens`, `reasoningTokens`) AND the v6+ nested fields (`inputTokenDetails.cacheReadTokens`, `inputTokenDetails.cacheWriteTokens`, `outputTokenDetails.reasoningTokens`). Cache-write tokens live ONLY under `inputTokenDetails.cacheWriteTokens` in v6+ — there is no deprecated top-level equivalent — and must map to `gen_ai.usage.cache_creation_input_tokens` (Anthropic prompt caching, Gemini context caching, etc.). Mirror this shape in tests: provider usage objects include `inputTokenDetails: { noCacheTokens, cacheReadTokens, cacheWriteTokens }` and `outputTokenDetails: { textTokens, reasoningTokens }`.

# Agents SDK — TS vs Python parity

- TS `ModelRequest` puts `systemInstructions` as a top-level field (not inside a kwargs bag like Python). The Python wrapper extracts it from `kwargs`; the TS wrapper reads `request.systemInstructions`.
- TS agents span_data uses private fields `_response` and `_input` (not `response` / `input`). The Python handlers read the public field names — when porting, check both.
- Manual instrumentation in tests: call `setTracingDisabled(false)` + `setTraceProcessors([])` first to clear any globally registered processors, then pass a `manuallyInstrument({ addTraceProcessor, OpenAIResponsesModel, OpenAIChatCompletionsModel })` shim that forwards to `setTraceProcessors([p])`.
- When mapping agents SDK `TracingProcessor` callbacks to Laminar spans, resolve parent spans via the agents SDK's `span.parentId` (looking up our `state.spans` map) and pass the result as `parentSpanContext`. Do NOT rely on `LaminarContextManager.getContext()` / `startActiveSpan({ global: true })` for child spans — the global stack is process-wide, so concurrent agent runs would cross-contaminate parent chains. Start from `ROOT_CONTEXT` to keep traces isolated.
- Processor-created spans use `Laminar.startSpan` (not `startActiveSpan`) so they skip Laminar's built-in activation path (which pops LIFO on `end()` and would corrupt the stack when agent spans end out of order). But to keep nested Laminar instrumentation (google-genai, etc.) running inside tool / agent callbacks parented to the agents span, the processor manually pushes the activation context onto the Laminar ALS stack on `onSpanStart` and removes it by reference (not LIFO) on `onSpanEnd` via `LaminarContextManager.removeContext(ctx)`. Reference-based removal is what makes out-of-order ends safe.
