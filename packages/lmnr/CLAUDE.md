This is the main `@lmnr-ai/lmnr` SDK package — OpenTelemetry-based tracing for the Laminar observability platform.

# Debug mode / replay (`src/debug/`) — server-cache v2

- A "debug run" is a normal process started with `LMNR_DEBUG*` env vars; there is no cache server, SSE, orchestrator, or bundling. In **v2 the replay cache lives server-side** (Redis), keyed per-LLM-call by the hex blake3 hash of the call's canonical-JSON input messages (system excluded). The SDK no longer fetches a source trace or builds an in-process cache, so `Laminar.initialize()` is fully synchronous and `initDebugRuntime(rolloutSessions, debuggerUrl=null)` returns just `{ runtime }` (no `ready` promise, no spine, no `ReplayCache`, no occurrence counters, no `awaitCacheReady`, no `source-trace.ts`).
- `src/debug/` is a **cross-language parity surface** with Python `src/lmnr/sdk/debug/`. Keep `config.ts`, `hash.ts`, `pointer.ts`, `replay.ts`, `index.ts` line-comparable; shared vectors (`test/data/debug/*.json`) stay byte-identical across repos. Truthy set stays `["true","1","yes","on"]`; `CONSOLE_PREFIX = "LMNR_DEBUG_RUN "`. **The persisted file is now `.lmnr/debug-session.json` (was `.lmnr/last-run.json`). Its CONTRACT — the `DebugSessionFile` interface + `DEBUG_SESSION_DIR`/`DEBUG_SESSION_FILE` consts — lives in `@lmnr-ai/types` (`packages/types/src/debug-session.ts`, type-only, no `node:fs`); the sync read/write helpers live in each consumer: the SDK's `src/debug/debug-session-file.ts` (`readDebugSessionFile` + `writeDebugSessionFile`) and the CLI's `src/utils/debug-session-file.ts` (`writeDebugSessionFile`).** Both import the shared interface + filename consts, which is what keeps the on-disk shape from drifting between the two writers (a previous version put the fs helpers in `@lmnr-ai/types`, which forced an `@types/node` devDep + `types:["node"]` into a type-only package — moved out). `DebugSessionFile` field order is `{session_id, trace_id, replay_trace_id, cache_until, debugger_url, started_at}` — `session_id` is now first AND primary (the field read at startup), and `trace_id` is `string | null` (a freshly-`new`-minted session has no trace yet). The TS-side file was renamed; the Python SDK still writes `last-run.json` — a Python follow-up will rename it (the file is local, not on the wire, so divergence is tolerable short-term).
- **`cache_until` is ALWAYS a span-id needle in v2** — the v1 count form is gone. `config.cacheUntilSpanId` is a hyphen-stripped lowercase hex string (`/^[0-9a-f]+$/`, `0 < len <= 32`) or null; `DebugSessionFile.cache_until` and `@lmnr-ai/types`' `cache_until` are now `string | null` (were `number`). `replayEnabledForConfig = replayTraceId !== null && cacheUntilSpanId !== null`. There is no client-side resolution step — the server matches the needle against its stored trace.
- **Per-call hashing (`hash.ts`).** `debugInputHash(messages) = hex(blake3(canonicalJson(stripSystemMessages(messages))))`. `canonicalJson` is a faithful port of Rust `canonical_json` (`app-server/src/traces/input_dedup.rs`): keys sorted lexicographically (recursive), arrays preserved, no whitespace, `undefined` dropped (serde skips `None`). KNOWN parity gap (shared with the v1 server): NO float canonicalization — `1.0` serializes as `1` in JS but `1.0` under serde. `stripSystemMessages` drops EVERY `role==="system"` message regardless of content or position (mirrors Rust `debug_input_hash` in `input_dedup.rs` — the coding agent may edit its own system prompt between iterations, so system content must never affect the cache key). The whole non-system array is hashed as ONE blob. Vectors in `test/data/debug/input_hash_cases.json` are a byte-identical mirror of the app-server's `app-server/test/data/debug/input_hash_vectors.json` (cases carry `input` + `expected_hash`); consumed by `test/debug/hash.test.ts`. (Python `debug/hash.py` still has the old first-non-empty strip semantics + an aisdk-normalize port — parity follow-up pending.)
- **What gets hashed is the VERBATIM LanguageModel-level prompt — there is no reshape layer (LAM-1922).** `aisdk-normalize.ts` is deleted. Both the recorded `gen_ai.input.messages` attribute and the replay-time hash flow through the SAME serializer, pinned by the shared helpers in `aisdk/utils.ts`: `verbatimPromptString(prompt)` (try/catch around AI SDK's `stringifyPromptForTelemetry` — the only transformation is Uint8Array file data → base64; `null` on failure) and `verbatimPromptMessages(prompt)` (`JSON.parse` of the former, `null` if not an array). The server stores/hashes these bytes verbatim too, so record-time key == replay-time key by construction. The end-to-end invariant is pinned in `test/debug/hash.test.ts` ("record-time vs replay-time hash equality"). Old-SDK-recorded traces (pre-verbatim `ai.prompt.messages` shapes) replayed by a new SDK simply MISS and degrade to live — acceptable, keys are self-consistent within a run.
- **The cache lookup runs per LLM call and degrades safely.** `DebugRuntime.lookupCache(inputHash)` returns `{ kind: "live" }` when replay isn't fully configured, else delegates to `RolloutSessionsResource.cache({ sessionId, replayTraceId, cacheUntil, inputHash })` (`POST /v1/rollouts/{sessionId}/cache`). `CacheOutcome` (exported from `@lmnr-ai/client`) is `{kind:"hit"; cached: CachedSpan} | {kind:"miss"} | {kind:"live"}`. The resource maps the server's `body.outcome` and degrades to `{kind:"live"}` on ANY transport error / non-OK status / parse failure / unknown outcome — it NEVER throws and NEVER latches the process-wide live flag. `toCachedSpan` is a tolerant mapper from the opaque HIT `response` payload onto the `CachedSpan` shape `parseCachedSpan` reads.
- **The ONLY replay consumer is `aisdk/base-language-model.ts`** (`doGenerateOrStreamWithCaching`). It guards on `replayEnabled() && !Laminar.debugRunLive`, hashes the reshaped input, then switches on the outcome: **HIT** → `markSpanCached(spanToMark)` + reconstruct the response from `outcome.cached` (never hits the wire); **MISS** → set the process-wide static `Laminar.debugRunLive = true` (every later call skips the lookup and runs live while the server warms the cache) + run live; **LIVE** → run THIS call live WITHOUT latching (server COLD warmup-timeout degrade, or any lookup error).
- **The CACHED-marked span must be the actual LLM span, and resolving it differs by AI SDK version.** `spanToMark = peekActiveLlmSpan() ?? Laminar.getCurrentSpan()`, captured **synchronously at the wrapper's `doGenerate`/`doStream` entry, before any await.** On **v7** the real `ai.llm` span is created by the telemetry integration (`v7-integration/index.ts`) and parked in `llmByKey` — it NEVER enters the OTel context, so `getCurrentSpan()` would resolve to the on-context top-level operation span (`ai.generateText`, a DEFAULT span) and mark the WRONG span. The integration therefore publishes each open LLM span to the tiny LIFO registry `aisdk/active-llm-span.ts` (`pushActiveLlmSpan` on `onLanguageModelCallStart`/`onObjectStepStart`; `removeActiveLlmSpan` on EVERY llm-span exit: `onLanguageModelCallEnd`, `onObjectStepFinish`, the `onStepFinish` defensive close, and the `endOrphanChildSpansForCallId` sweep). AI SDK fires `onLanguageModelCallStart` immediately before `model.doGenerate` with no await between, so the top of the stack is THIS call's LLM span. On **v6** the integration isn't used (registry empty), and the SDK activated the live `doGenerate` LLM span on the context via Laminar's tracer, so `getCurrentSpan()` IS the right span. `onLanguageModelCallEnd` never rewrites `SPAN_TYPE`, so the CACHED stamp set during the HIT survives to export. `active-llm-span.ts` is TS-only AI-SDK glue, NOT on the `src/debug/` parity surface. Tested in `test/active-llm-span.test.ts`.
- **A failed serialization must NOT be hashed.** `verbatimPromptMessages` returns `unknown[] | null` — `null` on a `stringifyPromptForTelemetry`/`JSON.parse` failure. The consumer (`base-language-model.ts`) treats `messages === null || messages.length === 0` as a can't-key signal and degrades to a live call WITHOUT latching (same as a transport error), BEFORE calling `debugInputHash`/`lookupCache`. Hashing a default/empty payload there would key off the wrong bytes, force a spurious MISS, and wrongly latch `debugRunLive` for the whole replay. (A system-only prompt still returns a length-1 array here and is stripped to `[]` inside `debugInputHash`, so that legit case is unaffected.)
- **`replayEnabled()` (`replay.ts`) gates on `getRuntime()?.replayConfigured ?? false`, NOT on `getRuntime() !== null`.** A debug-no-replay run (`LMNR_DEBUG` set but no replay trace / cache-until span id) still creates a runtime — needed to stamp `rollout.session_id` + emit the pointer — but replay stays off so the provider wrappers skip the per-call lookup entirely. `DebugRuntime.replayConfigured` is `replayEnabledForConfig(this._config)`. `markSpanCached(span)` sets `lmnr.span.type=CACHED` + `lmnr.span.original_type=LLM` (no-op when the span isn't recording / is null); `spanPathFromSpan` joins `lmnr.span.path` with `.`. Python mirrors `replay_configured` for line-parity.
- **The debug-session file location is resolved by walking UP from cwd to the nearest ancestor with a usable `.lmnr/debug-session.json`** (`findDebugSessionDir` / `resolveDebugSessionDir` in `debug-session-file.ts`; `readDebugSessionFile(dir)` itself stays an exact-dir read). Startup (`config.ts`) reads through `resolveDebugSessionDir()` and `emitPointer` writes back to the SAME anchor — read and write must agree or a run from a subdirectory would join the ancestor's session at init and then shadow it with a stale nested copy at shutdown. Falls back to cwd when no ancestor has a file. Matches the CLI's resolution (its `src/utils/debug-session-file.ts` mirrors these helpers). Python mirrors `find_debug_session_dir` / `resolve_debug_session_dir` in `debug_session_file.py` — same anchor in `config.py` + `pointer.py`, including the clobber guard.
- **`emitPointer` (pointer.ts) re-reads the on-disk session before writing and SKIPS the file write when the on-disk `session_id` differs from the record's** (the console marker is still always printed). The pointer is emitted at shutdown — possibly long after the run started — so an older `LMNR_DEBUG` process whose `exit` hook fires AFTER `lmnr-cli debug session new` reset `.lmnr/debug-session.json` would otherwise clobber the freshly-minted session id. Startup (`config.ts`) guarantees our in-memory `sessionId` matches what was on disk at init, so a divergence at shutdown means a newer session took ownership and we must not overwrite it.
- The record's **`started_at` is captured at `DebugRuntime` construction (SDK init), NOT inside `buildDebugSessionFile`/`emitPointer`** — the record is emitted at shutdown, so reading the clock there would stamp process-exit time. `DebugRuntime._startedAt` field initializer → `buildDebugSessionFile({ startedAt })`; the arg defaults to now only for standalone callers (pointer unit tests). Python mirrors via `DebugRuntime._started_at`.
- `initDebugRuntime` is one-shot via the module `initialized` flag, but the OFF path (null config, debug mode off) does NOT latch it — so a later init after the env flips `LMNR_DEBUG` on still initializes. `Laminar.shutdown()` MUST call `resetDebugRuntime()` (resets `runtime` + `initialized`) or a second `initialize()` resurrects the first run's stale session metadata / spent pointer and never re-reads `LMNR_DEBUG*`. Mirrors Python `reset_debug_runtime`.
- **The debug `exit` pointer hook must be removed on `shutdown()` or it leaks across init/shutdown cycles.** `_initDebugRuntime` registers `process.once("exit", () => runtime.emitPointer())`; `process.once` only auto-detaches AFTER `exit` fires, so a repeated `initialize()` (or init/shutdown loop) piles up listeners until Node logs `MaxListenersExceededWarning` (~10). Keep the handler by reference in static `Laminar.debugExitHook`, `removeListener` the prior hook before re-registering, and `removeListener` + null it in `shutdown()` (the pointer is already emitted synchronously at the top of `shutdown()`). `laminar.ts` is NOT part of the `src/debug/` parity surface. Tested in `test/initialize.test.ts` (`does not leak exit listeners across init/shutdown cycles`).
- **A debug run registers its SDK-minted session id with the backend** via `client.rolloutSessions.register({ sessionId })` (`POST /v1/rollouts/{sessionId}`, body `{ name }`, project-API-key auth). Without it a bare `LMNR_DEBUG=true` run mints a session id the backend never learns about, so the run is invisible in the UI. The call lives in `Laminar._initDebugRuntime` right after the `runtime === null` guard and is **fire-and-forget** (`void client.rolloutSessions.register(...).then(...).catch(...)` — synchronous `initialize()` must never block / throw). `register` returns `Promise<string | null>` (backend `projectId`, `null` on parse failure); the `.then` calls `runtime.recordProjectId(projectId)` (first-wins) then `logger.info(\`Laminar debugger session: ${runtime.debuggerSessionUrl()}\`)`. `debuggerSessionUrl()` is the **one shared URL path** used by both the console line AND the pointer's `debugger_url`: `${base}/project/${projectId}/debugger-sessions/${sessionId}`, falling back to the bare base when no project id was recorded, `null` when even the base is unset. Python mirrors `debugger_session_url()` / `record_project_id`. **After editing `register`/`cache` return types you MUST `pnpm --filter @lmnr-ai/client build` before linting / building `packages/lmnr`** — the consumer reads the *built* `@lmnr-ai/client` types, and a stale return makes the inference go wrong (e.g. `never` → `restrict-template-expressions` lint error). Only `register` (POST), the new `cache` (POST) and `delete` (DELETE) survive on the resource — the old SSE run-stream and PATCH span-update endpoints were removed. Tested in `packages/client/test/rollout-sessions-resource.test.ts`.
- **`.lmnr/debug-session.json` is now DEFAULT-ON — both the `LMNR_DEBUG_FROM_LAST_RUN` and `LMNR_DEBUG_WRITE_LAST_RUN_TO_FILE` env gates are DELETED.** `buildDebugConfig` always reads `readDebugSessionFile()` (`./debug-session-file`) and `emitPointer`/shutdown always writes via `writeDebugSessionFile`. The kept debug env vars are `LMNR_DEBUG`, `LMNR_DEBUG_SESSION_ID`, `LMNR_DEBUG_REPLAY_TRACE_ID`, `LMNR_DEBUG_CACHE_UNTIL`. The startup state machine: session id precedence `LMNR_DEBUG_SESSION_ID` → `existing.session_id` (file) → freshly-minted UUID; `sessionMinted` is true ONLY when we minted (so a **file-present run is a CONTINUATION — it rejoins the same session silently, no browser**, which is the core UX change). `replayTraceId` = `LMNR_DEBUG_REPLAY_TRACE_ID` → `existing.replay_trace_id` → null; `cacheUntilSpanId` = `parseCacheUntil(LMNR_DEBUG_CACHE_UNTIL ?? existing.cache_until)`. **Continuation ≠ replay**: unlike the old `FROM_LAST_RUN` path, the prior run's `trace_id` is NOT promoted into `replayTraceId` — replay is armed explicitly by the agent via the env var or the file's `replay_trace_id`. The CLI's `lmnr-cli debug session new` is what mints a fresh session + resets the file; the SDK only mints when NO file exists. Keep line-comparable with Python (pending the Python file rename). Tested in `test/debug/config.test.ts`.
- **A debug run propagates its coordinates downstream as ONE nested `debug` block of the serialized `LaminarSpanContext`** (`DebugContext` in `@lmnr-ai/types`: `enabled`, `sessionId?`, `replayTraceId?`, `cacheUntil?`). `getLaminarSpanContext()` (`opentelemetry-lib/tracing/span.ts`) stamps it via the module-local `currentDebugContext()` (reads `getRuntime()`, best-effort → `undefined` when debug is off; only `enabled:true` blocks are ever produced — we are the sole producer). `deserializeLaminarSpanContext` (`utils.ts`) parses it via `deserializeDebugContext` (accepting camelCase + snake_case; ids normalized via `normalizeUuidLike`, `cacheUntil` kept VERBATIM as the needle). Backward-compatible: old SDKs never send it, the block is usually absent in production. This is the **cross-language parity surface** — keep `buildDebugConfigFromContext` line-comparable with Python `build_debug_config_from_context`; `deserializeDebugContext` mirrors Python `DebugContext.deserialize`.
- **A downstream service arms — and then REFRESHES — its debug runtime from that propagated block**, so it joins the upstream run regardless of how its spans originate. `Laminar._armDebugRuntimeFromContext(debug)` is called from `_startSpan` after extracting the parent fields, before `tryToOtelSpanContext` — i.e. deep in span creation, BEFORE any caching instrumentation consults the runtime, AFTER the context is parsed. **The coordinates in a propagated `debug` block are DYNAMIC** — that is the entire point of carrying them on the span context. A long-lived downstream service handling many requests must follow each request's `sessionId` / `replayTraceId` / `cacheUntil`, NOT freeze on the first context it ever saw. So the design **separates the stable transport from the dynamic coordinates**: the `LaminarClient` / `rolloutSessions` handle is built once on the first arm and REUSED; the replay coordinates are refreshed in place on every subsequent context via `DebugRuntime.updateContextConfig(config)` (mutates only `sessionId`/`replayTraceId`/`cacheUntilSpanId`; `localOrigin`/`sessionMinted` are run IDENTITY and never change). `initDebugRuntimeFromContext` returns `{ runtime, sessionChanged }`: no runtime → build one (latch `initialized`); a `localOrigin:false` runtime exists → refresh coordinates + reuse client; a `localOrigin:true` (env) runtime exists → **env config still wins, untouched** (the local process owns the run; a propagated context never hijacks it). `_armDebugRuntimeFromContext` only (re-)registers the session + re-stamps `rollout.session_id` when `sessionChanged` is true, so a steady stream of requests on the same session does not spam the backend or rewrite global metadata per span. The from-context config is `localOrigin:false`: the downstream run reuses the upstream session, may consult the cache, stamps `rollout.session_id`, fire-and-forget registers the session (idempotent), but does NOT open the browser, log the session URL, or register a `process.once("exit")` pointer hook (the origin owns those). Because the from-context path has no access to `initialize()`'s args, `_initDebugRuntime` captures `baseUrl`/`httpPort` into static `Laminar.baseUrlForDebug` / `httpPortForDebug` (set unconditionally, even when local debug is off) so it can build its own `LaminarClient` later. **A from-context runtime can be armed BEFORE `initialize()`** (public `startSpan`/`startActiveSpan` have no `isInitialized` guard) — its cache `LaminarClient` is then built from still-unset connection args and falls back to env/defaults. So `_initDebugRuntime` ALSO **rebinds** that runtime's rollout-sessions handle once the real args are known (`runtime.rebindRolloutSessions(client.rolloutSessions)`, guarded on `!localOrigin`). `DebugRuntime._rolloutSessions` is therefore NOT `readonly`, and it exposes a `rolloutSessions` getter so the next arm can reuse the handle. Tested in `test/debug/runtime.test.ts` (`updateContextConfig …`, `initDebugRuntimeFromContext refreshes a context runtime …`, `… never overrides an env-origin runtime`, `rebindRolloutSessions …`) and `test/span.test.ts` (`refreshes the downstream session id when a new context arrives`). Mirrored in Python `_arm_debug_runtime_from_context` / `init_debug_runtime_from_context` / `DebugRuntime.update_context_config`.
- **The one-time browser open is gated on the resolved config, not env vars.** `DebugConfig` gained `localOrigin` (origin vs inherited) and `sessionMinted` (fresh vs reused/provided session id); `DebugRuntime.shouldOpenBrowser` is `localOrigin && sessionMinted`. `_initDebugRuntime` opens the browser on `if (runtime.shouldOpenBrowser)` — a reused session id (explicit `LMNR_DEBUG_SESSION_ID` OR a continuation read from `.lmnr/debug-session.json`) or a context-armed downstream run no longer reopens the browser. `DebugRuntime` also gained a `cacheUntilSpanId` getter (needed by the producer `currentDebugContext`). Tested in `test/debug/config.test.ts` (DebugContext parse, from-context builder, local-origin/session-minted, nested-debug `LaminarSpanContext` parse).
- Naming: new env/file/module naming is `debug`; the persisted metadata key stays `rollout.session_id` (intentional — do not rename to `debug.session_id`).
- **The run pointer's `trace_id` is recorded from two places**, because not every debug run opens a root span. Normal path: `LaminarSpanProcessor.onStart` → `recordDebugTraceId` when `parentSpanId === undefined`. But an `LMNR_SPAN_CONTEXT`-attached run pushes a wrapped non-recording parent onto the context, so the root hook never fires — `_initializeContextFromEnv` therefore also calls `getRuntime()?.recordTraceId(otelTraceIdToUUID(otelSpanContext.traceId))` (first wins). `_initDebugRuntime` runs before `_initializeContextFromEnv` in `initialize()`. Python mirrors via `Laminar._record_debug_trace_id_from_env`.
- **The debug run's `rollout.session_id` reaches spans via global metadata, which `LaminarSpanProcessor.onStart` stamps on EVERY span.** TS does not carry global metadata on the OTEL context — `setGlobalMetadata` only stores a static field on `LaminarContextManager` — so `onStart` reads `getGlobalMetadata()` + `metadataToAttributes(...)` directly (BEFORE the association-properties block so per-span metadata still wins); relying on the context block alone would lose `rollout.session_id` on pure auto-instrumentation traces. Python gets this free via `__global_metadata` on the ambient context.

# Git metadata (`src/git-metadata.ts`)

- `Laminar.initialize()` auto-collects `git.commit` / `git.branch` / `git.dirty` (best-effort `execSync` with a 1.5s timeout, guarded by `process?.versions?.node`; CI env-var fallback: `GITHUB_SHA`, `VERCEL_GIT_COMMIT_SHA`, etc.) and merges it into `globalMetadata` at the LOWEST precedence — `LMNR_TRACE_METADATA` and the `metadata` init arg both override git keys. Opt out via `disableGitMetadata: true` or `LMNR_DISABLE_GIT_METADATA`. Eval runs get the same keys in their run metadata via `withGitMetadata` (composed with `withSessionMetadata` at the `client.evals.init` call site in `evaluations.ts`).
- `collectGitMetadata()` caches its first result for the process lifetime in a module-level variable; tests that vary cwd/env MUST call `resetGitMetadataCache()` (the `test/git-metadata.test.ts` beforeEach does, plus `setGitMetadataDisabled(false)`). The disabled check (init opt-out via `setGitMetadataDisabled` OR the env var) runs OUTSIDE the cache, on every call, so the `disableGitMetadata: true` opt-out recorded by `initialize()` also suppresses eval-run metadata collected later — Bugbot finding on PR #280; mirrors the Python fix on PR #311.
- Cross-language parity surface with `lmnr-python/src/lmnr/sdk/git_metadata.py` — keep key names, CI env-var list, precedence, and the truthy set line-comparable.
- `git.dirty` uses `git status --porcelain --untracked-files=no` (tracked changes only); `git.branch` is omitted on a detached HEAD (`rev-parse --abbrev-ref HEAD` returning literal `HEAD`).
- Tests that spawn `git` in `mkdtempSync` temp dirs set `GIT_CEILING_DIRECTORIES` so git discovery can't walk up into a real repo.

# Tests

- Tests are run with `pnpm test` which invokes `tsx --test test/*.test.ts test/**/*.test.ts`. No Jest, no Vitest.
- HTTP calls are mocked via `nock` using cassette recordings under `test/recordings/*.json`. `decompressRecordingResponse` (in `test/utils.ts`) handles gzipped payloads; hand-crafted cassettes just set a plain JSON `response`.
- `initializeTracing({ exporter, disableBatch: true })` + `Object.defineProperty(Laminar, "isInitialized", { value: true, writable: true })` is the standard test setup for asserting on exported spans via `InMemorySpanExporter`.
- **`emitPointer()` writes `${cwd}/.lmnr/debug-session.json` as a best-effort side effect (default-on now — no env gate).** The `debug/pointer.test.ts` / `debug/runtime.test.ts` cases that assert on the *file* override `process.cwd` to a `mkdtempSync` temp dir, but the ones that only assert on the *console line* (`recordTraceId keeps the first trace id` in `runtime.test.ts`, the `LMNR_SPAN_CONTEXT` case in `initialize.test.ts`) do NOT — so they leak a real `.lmnr/` into the package dir. Both suites guard this with a `before()` that records `existsSync(.lmnr)` and an `after()` that `rmSync`s the dir ONLY if this run created it (never clobber a pre-existing one). When adding a debug test that emits a pointer, either override `process.cwd` to a temp dir or rely on this guard.

# Context / startActiveSpan

- `Laminar.startActiveSpan` activates the span on `LaminarContextManager`'s AsyncLocalStorage stack. Sibling / unrelated async tasks (i.e. callbacks that run outside the returned promise's chain — `setImmediate`, separate timers, independent processor callbacks, etc.) do NOT inherit the activated span — they live in a different ALS frame. For spans that must parent across detached async callbacks, pass `global: true`: the span is pushed onto a process-global stack that `getContext()` falls back to when the ALS stack is empty. The span is popped from that stack automatically in `LaminarSpan.end()`. Use sparingly — it is roughly analogous to Python's `context.attach(...)` / `start_as_current_span` global activation.
- If you pass `context: ...` into `startSpan` / `startActiveSpan` it replaces the usual `LaminarContextManager.getContext()` base — so if you need the global fallback, build your context on top of `LaminarContextManager.getContext()` rather than on `otelContext.active()`.
- `LaminarSpan.end()` only calls `popContext()` once (for the activated-span push). Do NOT push additional context frames elsewhere in `_startSpan` without a matching pop — `popContext`'s filter drops _already-ended_ spans by spanId, but a leaked wrapper of a still-live parent span (e.g. `trace.wrapSpanContext(...)` built from `parentSpanContext`) looks "valid" and stays on the stack, growing it unboundedly across repeated start/end pairs. The `parentSpanContext` path is resolved purely via `trace.setSpan(entityContext, ...)` + passing `entityContext` to `getTracer().startSpan(..., entityContext)` — no push needed.

# OpenTelemetry Instrumentations

- Instrumentations live under `src/opentelemetry-lib/instrumentation/<name>/` and are registered in `src/opentelemetry-lib/tracing/instrumentations.ts` (both auto-registration path and the `manuallyInitInstrumentations` path — keep the two in sync).
- For manual instrumentation, expose an option key on `InitializeOptions.instrumentModules` and wire it through `manuallyInitInstrumentations`.
- `TracingProcessor` from `@openai/agents` requires the interface methods return `Promise`. Most of our methods have no `await` — add a file-level `/* eslint-disable @typescript-eslint/require-await */` around the class rather than sprinkling `await Promise.resolve()`.
- `@openai/agents` re-exports types and classes from `@openai/agents-core` and `@openai/agents-openai`. Import from `@openai/agents` to avoid adding `-core`/`-openai` as direct dependencies.
- When patching prototypes registered via multiple `InstrumentationNodeModuleDefinition` entries that resolve to the same class (e.g. `@openai/agents` and `@openai/agents-openai` both expose `OpenAIResponsesModel.prototype`), guard `_wrap` with a `WeakSet` of already-wrapped prototypes — `shimmer` happily double-wraps and will run your wrapper logic twice per call.
- `AsyncLocalStorage.run(value, fn)` restores the previous context as soon as `fn` returns — for async iterables/streams, wrap **each** `next()`/`return()`/`throw()` call in its own `.run()` so the context is visible across yields, not just at iterator creation.

# Stagehand v3 Instrumentation (`src/browser/stagehand/v3/`)

- `page.sendCDP(method, params)` was added in `@browserbasehq/stagehand` **3.0.2** (2025-11-14). Only **3.0.0** (2025-10-29) and **3.0.1** (2025-10-31) lack it — on those versions Page's `mainSession` is private and there is no `sendCDP`, so CDP calls must go through `page.getSessionForFrame(page.mainFrameId()).send(method, params)` (both methods are public on 3.0.0/3.0.1 Page). Use the `sendCDP(page, method, params)` helper in `cdp-helpers.ts` — it probes `page.sendCDP` first, falls back to `getSessionForFrame`, and throws if neither exists. Without this fallback, recording injection silently fails on 3.0.0/3.0.1 with `page.sendCDP is not a function` and no rrweb events reach the backend even though spans still flow. Stagehand PR #983 (merged pre-3.0.2) made `sendCDP` public specifically for external CDP-based instrumentations like ours; the v3 instrumentation itself landed Dec 2025 when 3.0.5 was latest, so 3.0.0/3.0.1 are the only versions that were ever shipped without the needed API.
- The `@lmnr/browserbase-stagehand-instrumentation` instrumentation declares `['>=3.0.0']` as the supported range, so 3.0.0, 3.0.1, 3.0.2+, and 3.1.x+ must all keep working — test any CDP-touching change against at least one pre-sendCDP version (3.0.1) and one post-sendCDP version (3.1.0).

# AI SDK v7 Telemetry (`src/opentelemetry-lib/instrumentation/aisdk/v7.ts`)

- AI SDK v7 replaced the `experimental_telemetry: { tracer }` wire-a-tracer API with a callback-based `Telemetry` integration (16 optional lifecycle callbacks + `executeTool`) passed as `telemetry: { integrations: [...] }`. Both `telemetry` and `experimental_telemetry` keys are accepted on the call site, but ONLY v7 understands `integrations`; v6 still expects a `tracer`. Detect v7 by `typeof ai.registerTelemetry === 'function'` (added only in v7) and branch `wrapAISDK` accordingly — don't sniff by version string. On v7 the wrapper MUST NOT pass `tracer`, and on v6 it MUST NOT pass `integrations`.
- Span hierarchy is `operation (ai.generateText|ai.streamText|ai.generateObject|ai.streamObject)` → `step (ai.step N)` → flat siblings `ai.llm <model>` and `ai.tool <name>`. Tool spans are NOT children of the LLM span even though tool calls are emitted from LLM content — they're siblings under the same step. Embed/rerank (`ai.embed`, `ai.embedMany`, `ai.rerank`) get a SINGLE LLM span at the operation level with no step/llm children; branch on `operationId` inside `onStart` to set `SPAN_TYPE = LLM` directly. **Attribute strategy is verbatim (LAM-1922)**: LLM spans carry `gen_ai.input.messages` (the LanguageModel-level prompt through `verbatimPromptString` — see the stash bullet below) + `gen_ai.output.messages` (`[{role:"assistant", content: event.content}]` — the provider's `LanguageModelContent` untouched). `ai.prompt.messages`, `ai.response.text`, and `ai.response.toolCalls` are NO LONGER written anywhere. DEFAULT step/operation spans carry `SPAN_INPUT = verbatimStandardizedMessages(event)` (system prompt becomes a leading system message, `messages` appended UNCHANGED) and `SPAN_OUTPUT = buildVerbatimOutputMessages(event.content)`. Metadata keys (`gen_ai.system`, `gen_ai.request.model`, `gen_ai.response.id`, `gen_ai.response.finish_reason`, `gen_ai.usage.*`, `ai.model.provider`, `ai.model.id`, `ai.response.finishReason`) are unchanged. The Mastra exporter is OUT OF SCOPE — it keeps its old shapes (see root CLAUDE.md).
- **The LanguageModel-level prompt is stashed in `onStepStart` because `onLanguageModelCallStart` doesn't carry it.** `onStepStart` ALWAYS stashes `verbatimPromptString(event.promptMessages)` into `promptByCallId` keyed by **callId ALONE** BEFORE the `createStepSpan` early return (AI SDK fires `onLanguageModelCallStart` immediately after with no await, but that event only has ModelMessage-level fields). Do NOT key by `stepKey(callId, stepNumber)`: v7's `languageModelCallStartEvent` carries NO `stepNumber` (verified against ai@7.0.17 `dist/index.js` — only `onStepStart` events have it), so with `createStepSpan=false` (empty `stepByKey`) a stepKey lookup resolves `callId:0` while step N stashed under `callId:N` — the stash misses on every step after the first. Steps within one callId are strictly sequential, so one slot per callId is unambiguous. `onLanguageModelCallStart` reads the stash for `gen_ai.input.messages` and **OMITS the attribute when there is no stash — there is deliberately NO `verbatimStandardizedMessages(event)` fallback**: those ModelMessage-level bytes differ structurally from the LanguageModel-level `options.prompt` the replay wrapper hashes (e.g. raw string user content stays a string at ModelMessage level but is a `[{type:"text",text}]` parts array at LanguageModel level per `convertToLanguageModelMessage`), so a fallback-recorded input can NEVER hash-match `debugInputHash` at replay time — the cache MISSes and `debugRunLive` latches, degrading the whole debug run to live. A missing attribute is the safer failure mode (that call replays live without latching). `onObjectStepStart` reads `event.promptMessages` directly. The stash is cleaned in `onLanguageModelCallEnd`, `onStepFinish`, and the `endOrphanChildSpansForCallId` sweep — miss one and it leaks per operation.
- Sub-agent nesting (nested `generateText` inside `executeTool`) requires propagating the tool span as parent through BOTH OTel's standard context AND Laminar's ALS: wrap the user's `execute` with `contextApi.with(toolSpanCtx, () => LaminarContextManager.runWithIsolatedContext([...currentStack, toolSpanCtx], execute))`. OTel-aware instrumentations read `contextApi.active()`; Laminar's own `startActiveSpan` only reads its ALS stack — miss either and the inner operation span parents to the outer operation instead of the tool span, breaking sub-agent traces.
- Per `wrapAISDK()` call create EXACTLY ONE `laminarTelemetry()` integration and share it across all wrapped AI functions — v7 dedupes integrations by reference, and a fresh integration per call would reset the internal `callId → spanRef` maps between `generateText` invocations in the same process. `ToolExecutionEndEvent` is a discriminated union on `toolOutput.type`: `"tool-result"` → set `ai.toolCall.output`; `"tool-error"` → `span.setStatus({ code: ERROR })` + `ai.toolCall.error = String(error)`.
- `toolByCallId` is keyed by `toolCallId` (AI SDK guarantees uniqueness per run), NOT by `${callId}:...` like `stepByKey` / `llmByKey`. Because of that, `onFinish` / `onError` cleanup sweeps CANNOT use `key.startsWith(prefix)` to scope tool spans to a single operation — `ToolState` must carry the owning `callId` so the sweep can filter by `tool.callId === targetCallId`. Without this, an `onError` mid-tool-execution leaves the tool span open indefinitely (unterminated span in the backend + map leak).
- In `onFinish` and `onError`, defensive cleanup of orphan child spans (llm / step / tool) must run BEFORE `op.span.end()`. Otherwise the parent operation span gets a stale end timestamp and the children end afterwards — breaking trace hierarchy semantics where every child must end before its parent. `onError` already did this; `onFinish` was ending the op first. The cleanup order within the child sweep also matters: tool spans are children of step spans (see `onToolExecutionStart` where `parentCtx = step?.ctx ?? op.ctx`), so the sweep order must be `llm + tool (leaves under step) → step → op`, NOT `llm → step → tool → op`. A test that only validates "children end before operation" does not catch this mid-level ordering — assert tool.endTime ≤ step.endTime explicitly.
- `SPAN_OUTPUT` on the operation span in `onFinish` is `buildVerbatimOutputMessages(event.content)` — tool-call-only single-step responses are covered because `event.content` includes the tool-call parts verbatim (the backend reads `SPAN_OUTPUT` for the span output panel).
- **LLM-span output is `gen_ai.output.messages` ONLY — no legacy `ai.response.text` / `ai.response.toolCalls` / `SPAN_OUTPUT` on the LLM span.** Every LLM-span output write goes through `v7-integration/utils.ts`: `buildVerbatimOutputMessages(event.content)` emits `[{role:"assistant", content}]` with the provider's content parts UNTOUCHED (`onLanguageModelCallEnd`, `onEnd`, step SPAN_OUTPUT); paths with no provider content (`onStepFinish` + `endOrphanChildSpansForCallId` buffered-stream fallbacks, `onObjectStepFinish`'s `objectText`) use `buildTextOutputMessages(text)` which wraps a single `{type:"text", text}` part in the same shape. The backend stores `gen_ai.output.messages` VERBATIM but ONLY on LLM spans (gated by `is_llm_span()` in `app-server/src/traces/spans.rs`). The **debugger replay consumer (`base-language-model.ts::convertToContentBlocks`) rebuilds the cached response from whatever the LLM span stored in ClickHouse**, so it handles THREE shapes: verbatim v7 `{role, content:[{type:"text"|"reasoning", text} | {type:"tool-call", toolCallId, toolName, input}]}` (tool-call `input` is usually ALREADY a JSON string — pass it through, never re-stringify, or replay double-encodes arguments), the older v7-hybrid `{role, parts:[{type, content} | {type:"tool_call", id, name, arguments}]}`, and the v6 legacy `{role, content}` with `tool_call` parts; keep all three paths working. Tested in `test/cached-stream.test.ts` (`rebuilds content from verbatim v7 ...` / `... v7 gen_ai.output.messages parts` / `... v6 legacy ...`).
- **The replay rebuild must be VERBATIM, not field-by-field.** `convertToContentBlocks` SPREADS verbatim-v7 parts through (preserving `providerMetadata`, `providerExecuted`, and any future provider fields), and `createStreamFromCachedResponse` in v2/v3/v4 threads `providerMetadata` onto `text-start`/`reasoning-start` and spreads the whole block into the `tool-call` stream part. Why: the AI SDK's `toResponseMessages` echoes each response part's `providerMetadata` into the NEXT step's prompt as `providerOptions` (tool-results inherit the tool-call's providerMetadata too, via `executeToolCall`), `convertToLanguageModelMessage` threads `providerOptions` through on every part type, and the server hashed the record-run step-N+1 input WITH those fields — so a rebuild that drops them (e.g. Anthropic's `{anthropic:{caller:{type:"direct"}}}`) makes the replayed step-N+1 prompt hash to different bytes → spurious MISS → `Laminar.debugRunLive` latches for the whole run. Legacy shapes (`tool_call` with `id`/`name`/`arguments`, `thinking`/`text` with `content`) must NOT be spread — their legacy keys would leak into the rebuilt part — so they keep the field-by-field rebuild. The v7 stream transform (`upsertTextContentPart`, tool-call chunk parsing) preserves chunk `providerMetadata` into `modelCallContent`, so the stream-part threading round-trips. Tested in `cached-stream.test.ts` (`preserves provider fields ...` / `threads provider fields onto rebuilt stream parts ...`).
- Sweep ordering alone is NOT enough to keep child.endTime ≤ parent.endTime deterministically. Four rapid synchronous `span.end()` calls inside `endOrphanChildSpansForCallId` read `performance.now()` / hrtime separately, and consecutive readings on the same tick are not strictly monotonic — even when they are, the test's f64 conversion (`sec * 1e9 + nanos` ≈ 1.78e18 today, >1000× past `Number.MAX_SAFE_INTEGER`) rounds both ends to the same bucket and flips the order. Fix: accumulate the maximum child endTime seen so far and explicitly pass it into `step.span.end(latest)` and `op.span.end(latest)`. The value MUST come from the underlying SDK span's `endTime` field (mutated by `end()`), NOT from `LaminarSpan.endTime` — that wrapper copies the field in its constructor before `end()` has set it, so reading it yields the `[0, 0]` sentinel. See the `readSpanEndTime` helper in `v7.ts` which probes `_span.endTime` first and falls back defensively.
- Tests drive the class with synthetic event payloads (see `test/aisdk-v7-telemetry.test.ts`'s `mkStartEvent` / `mkLlmCallStart` / `mkStepEnd` helpers) — no runtime dep on `ai` is needed and no real HTTP cassettes. Span attributes typed as `any` in callback payloads need a file-level `/* eslint-disable @typescript-eslint/no-explicit-any, @typescript-eslint/no-unsafe-argument */` on `v7.ts` rather than per-line disables.
- `recordOutputs: false` gates user-visible _content_ only (the LLM span's `gen_ai.output.messages`, and on step/operation spans `ai.rerank.ranking`, embedding shape, serialized `SPAN_OUTPUT`). The `gen_ai.response.finish_reason` / `ai.response.finishReason` metadata must be emitted regardless of `recordOutputs`.
- **Token usage (`applyUsageToSpan` / `gen_ai.usage.*`) is written ONLY on LLM-typed spans, never on the DEFAULT step or operation spans (LAM-1840).** The backend computes cost from `gen_ai.usage.*` on ANY span carrying them — it is NOT gated by `SPAN_TYPE` (see `app-server/src/traces/utils.rs`). So stamping the already-aggregated `event.totalUsage` on the DEFAULT operation span (`onEnd`) or the DEFAULT step span (`onStepFinish`) double-counts: the parent total gets summed on top of its LLM-leaf children, roughly 2× the trace cost. Usage therefore lives only on: the LLM leaf spans (`onLanguageModelCallEnd` + the `onStepFinish` orphan-close fallback — when `onLanguageModelCallEnd` was skipped mid-stream/on-error it deletes the map key, so `onStepFinish` writes `event.usage` onto the still-open LLM span to preserve cost), the object-gen LLM span (`onObjectStepFinish`), and the `ai.embed` / `ai.embedMany` / `ai.rerank` operation spans (which are classified LLM in `onStart` and have no LLM-leaf children, so `onEnd` writes usage on them gated by `op.operationId`). The parent operation total is still recoverable in the UI as the subtree rollup of its children. Regression pinned in `test/aisdk-v7-telemetry.test.ts` ("creates operation → step → llm span tree for generateText" asserts `gen_ai.usage.*` is `undefined` on both the op and step spans).
- `onChunk` in AI SDK v7 streams only the lifecycle markers `ai.stream.firstChunk` / `ai.stream.finish` carry `callId` + `stepNumber` — regular content chunks (`text-delta`, `tool-call`, ...) are plain `TextStreamPart` objects with only `type` / `id` / `text` (or equivalent), and carry NO callId either at top level or on the chunk. See AI SDK source `packages/ai/src/generate-text/stream-text.ts` — the dispatcher publishes the lifecycle chunks at the step boundaries and content chunks in between. To accumulate streaming text correctly, latch the active streaming step on `firstChunk`, route intermediate `text-delta` chunks to its LLM span, and clear the latch on `finish` (and on `onFinish` / `onError` cleanup paths to avoid leaks on aborted streams). An integration that requires callId/stepNumber on every chunk silently drops all text-delta chunks for streamText — buffered text ends up empty.
- Because content chunks carry no callId, the "latched active stream" set can hold more than one entry when 2+ concurrent `streamText` / `streamObject` calls share the same `LaminarTelemetry` instance. A LIFO / most-recently-inserted pick misattributes A's chunks to B as soon as B's `firstChunk` fires while A is still active. The correct behavior is to attribute only when the active set has exactly ONE entry — otherwise leave the chunk unattributed. The buffered-deltas path is only a fallback for aborted / errored streams where `onLanguageModelCallEnd` never fires; the normal path reads text from `event.content` which is correctly scoped per call, so declining to attribute ambiguous chunks costs little. The concurrent-stream regression test in `test/aisdk-v7-telemetry.test.ts` pins this behavior — do not replace the size-1 guard with a LIFO pick.
- `onObjectStepStart` / `onObjectStepFinish` fire ONLY from `generateObject` / `streamObject` (which call `model.doGenerate` / `doStream` directly). `onLanguageModelCallStart` / `onLanguageModelCallEnd` fire ONLY from `generateText` / `streamText` via `stream-language-model-call.ts`. The two callback pairs are mutually exclusive per operation — they never both fire for the same `callId`. Confirmed against vercel/ai `packages/ai/src/generate-object/generate-object.ts:352` + `stream-object.ts:584` (no `onLanguageModelCall*` dispatch) vs. `packages/ai/src/generate-text/stream-language-model-call.ts:295` (the only dispatch sites for `onLanguageModelCall*`). Keying the object-gen LLM span under `stepKey(callId, 0)` is therefore collision-free with `onLanguageModelCallStart`'s default `stepNumber = 0` fallback — no separate namespace needed.
- Object-gen LLM spans must carry attribute parity with the `generateText` LLM path — specifically the minimum-set attributes `gen_ai.request.model` (REQUEST_MODEL) + `gen_ai.response.model` (RESPONSE_MODEL) the backend uses for model-routing and cost analytics. `GenerateObjectStepEndEvent.response` (in `vercel-ai/packages/ai/src/generate-object/structured-output-events.ts`) carries `{ id, modelId, timestamp }`, so `onObjectStepFinish` must pull `gen_ai.response.model` from `event.response.modelId` (not `event.modelId`, which is the request model) and `gen_ai.response.id` from `event.response.id`. In `onObjectStepStart` also set the legacy `ai.model.provider` + `ai.model.id` attributes to mirror `onLanguageModelCallStart` — dashboards keyed off those specific attribute names render object-gen spans uniformly with text-gen spans.
- AI SDK v6+ `LanguageModelUsage` moved detail fields under `inputTokenDetails` / `outputTokenDetails`. `applyUsage()` must read both the deprecated top-level fields (`cachedInputTokens`, `reasoningTokens`) AND the v6+ nested fields (`inputTokenDetails.cacheReadTokens`, `inputTokenDetails.cacheWriteTokens`, `outputTokenDetails.reasoningTokens`). Cache-write tokens live ONLY under `inputTokenDetails.cacheWriteTokens` in v6+ — there is no deprecated top-level equivalent — and must map to `gen_ai.usage.cache_creation_input_tokens` (Anthropic prompt caching, Gemini context caching, etc.). Mirror this shape in tests: provider usage objects include `inputTokenDetails: { noCacheTokens, cacheReadTokens, cacheWriteTokens }` and `outputTokenDetails: { textTokens, reasoningTokens }`.

# OpenCode Instrumentation (`src/opentelemetry-lib/instrumentation/opencode.ts`)

- Trace continuity across the OpenCode subprocess boundary works by injecting a synthetic prompt part with `metadata.lmnrSpanContext` (a serialized `LaminarSpanContext`) in the patched `Session.prototype.prompt` / `promptAsync`. The OpenCode plugin (`@lmnr-ai/opencode-plugin`, separate repo `lmnr-ai/lmnr-opencode-plugin`) reads it from `chat.message`, strips it from `output.parts`, and creates the top-level `opencode turn` span with `parentSpanContext` set to that serialized context. The two halves run in different Node processes — there is no shared OTel context, only the on-the-wire prompt parts.
- Patching targets `Session.prototype.prompt` (and `promptAsync`), discovered by walking `OpencodeClient` → `instance.session.constructor`. `session` is a class field initialized in the constructor (not on the prototype), so `resolveSessionClass` MUST `new ClientClass()` to trigger field init before reading `instance.session.constructor`.
- The plugin lives in a separate repo and is fetched by OpenCode at runtime via `Npm.add` and cached under `~/.cache/opencode/packages/@lmnr-ai/opencode-plugin@latest/...` — opencode does NOT use the host project's `node_modules`. To test plugin changes locally, build the plugin and copy `dist/*` into that cache directory; reinstalling under `node_modules` has no effect on what opencode loads. The cache is only refreshed when the package version changes on npm, so a fresh `pnpm publish` is required for users to pick up plugin fixes (the SDK side and plugin ship independently).

# Agents SDK — TS vs Python parity

- TS `ModelRequest` puts `systemInstructions` as a top-level field (not inside a kwargs bag like Python). The Python wrapper extracts it from `kwargs`; the TS wrapper reads `request.systemInstructions`.
- TS agents span_data uses private fields `_response` and `_input` (not `response` / `input`). The Python handlers read the public field names — when porting, check both.
- Manual instrumentation in tests: call `setTracingDisabled(false)` + `setTraceProcessors([])` first to clear any globally registered processors, then pass a `manuallyInstrument({ addTraceProcessor, OpenAIResponsesModel, OpenAIChatCompletionsModel })` shim that forwards to `setTraceProcessors([p])`.
- When mapping agents SDK `TracingProcessor` callbacks to Laminar spans, resolve parent spans via the agents SDK's `span.parentId` (looking up our `state.spans` map) and pass the result as `parentSpanContext`. Do NOT rely on `LaminarContextManager.getContext()` / `startActiveSpan({ global: true })` for child spans — the global stack is process-wide, so concurrent agent runs would cross-contaminate parent chains. Start from `ROOT_CONTEXT` to keep traces isolated.
- Processor-created spans use `Laminar.startSpan` (not `startActiveSpan`) so they skip Laminar's built-in activation path (which pops LIFO on `end()` and would corrupt the stack when agent spans end out of order). But to keep nested Laminar instrumentation (google-genai, etc.) running inside tool / agent callbacks parented to the agents span, the processor manually pushes the activation context onto the Laminar ALS stack on `onSpanStart` and removes it by reference (not LIFO) on `onSpanEnd` via `LaminarContextManager.removeContext(ctx)`. Reference-based removal is what makes out-of-order ends safe.
